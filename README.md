# ✍️ Laravel Blog Project (Learning Track)

Проект персонального блога, созданный в процессе перехода с CMS-разработки (Bitrix/WP) на фреймворк **Laravel 9**. 
Здесь я документирую свой путь обучения, применяя архитектурные стандарты и лучшие практики PHP.

---

## 📈 Дневник разработки и освоенные технологии:

*(Здесь будут появляться пункты по мере прохождения курса)*

### 🛠 Шаг 1: Окружение и Базовая Авторизация
* **Инфраструктура:** Развернута локальная среда разработки с использованием **Docker**. Настроена и подключена база данных **MySQL**.
* **Frontend Scaffolding:** Интегрирован **Bootstrap** через пакет `laravel/ui` для быстрого прототипирования интерфейса.
* **Authentication:** Реализована базовая система регистрации и входа пользователей с помощью `php artisan ui:auth`.
* **Environment:** Настроена конфигурация `.env` для корректной работы приложения с контейнеризированной БД.

### 🗄 Шаг 2: Проектирование архитектуры БД (Database Schema)
* **Миграции:** Создана структура таблиц для контентной части блога: `posts` (посты), `categories` (категории) и `tags` (теги).
* **Целостность данных (Data Integrity):** * На уровне СУБД прописаны внешние ключи (**Foreign Keys**) и индексы.
    * Реализована каскадная логика связи для категорий (One-to-Many на уровне SQL).
    * Создана связующая таблица (Pivot table) для реализации связи Many-to-Many между постами и тегами.

### 🖼 Шаг 3: MVC и Шаблонизация (Blade)
* **Controller Layer:** Создан `MainController` для обработки запросов главной страницы блога.
* **Blade Layouts:** Реализована архитектура мастер-шаблонов (`Layouts`).
    * Создан базовый слой `layouts.main`, включающий общие элементы (Header, Footer).
    * Использованы директивы `@yield` и `@extends` для внедрения контента из дочерних представлений (`index.blade.php`).

### 🛡 Шаг 4: Административная панель (AdminLTE Integration)
* **Routing & Architecture:** Реализована группа маршрутов с префиксом `/admin` и пространством имен (`Namespace`) `App\Http\Controllers\Admin`.
* **Dashboard Setup:** Интегрирован шаблон **AdminLTE** для панели управления.
* **Asset Management:** Настроено подключение статичных ресурсов (CSS/JS) через хелпер `asset()`.
* **Layout Separation:** Создан отдельный мастер-шаблон для админки, изолированный от публичной части сайта. Очищена структура шаблона от избыточных компонентов для повышения производительности.

### 🏗 Шаг 5: Архитектура Single Action Controllers (CRUD - Create)
* **Design Patterns:** Применен подход **Single Action Controllers**. Логика разделена на узкоспециализированные классы:
    * `CreateController` — отвечает исключительно за отображение формы.
    * `StoreController` — отвечает только за обработку данных и сохранение в БД.
* **Invokable Methods:** Использован магический метод `__invoke()`, что делает контроллеры более лаконичными и упрощает роутинг.
* **Forms & Security:** Реализована страница добавления категории (AdminLTE) с обязательной защитой через `@csrf` токен.
* **Data Persistence:** Обработка входящего запроса и создание записи в таблице `categories`.

### 📋 Шаг 6: Вывод данных (CRUD - Read)
* **Logic:** Реализован `IndexController` (Single Action) для получения всех категорий из базы данных.
* **Eloquent:** Использован метод `Category::all()` для извлечения коллекции записей.
* **Blade Templating:** * Создана страница списка категорий с использованием таблиц AdminLTE.
    * Реализован динамический вывод данных через директиву `@foreach`.
    * Настроена нумерация и отображение ID/названий категорий.

### 🔍 Шаг 7: Детальный просмотр (CRUD - Read/Show)
* **Logic:** Реализован `ShowController` для отображения подробной информации о конкретной категории.
* **Routing:** Настроен динамический маршрут `admin/categories/{category}`.
* **UI & UX:** * Создана страница просмотра с использованием карточек AdminLTE.
    * Реализована навигация: возможность перейти к просмотру прямо из общего списка категорий.
* **Data Access:** Получение модели категории по её ID и передача данных в Blade-представление.

### ✏️ Шаг 8: Обновление данных (CRUD - Update)
* **Architecture:** Реализована пара однометодных контроллеров для процесса редактирования:
    * `EditController` — загрузка существующей модели и отображение формы редактирования.
    * `UpdateController` — обработка входящих данных и синхронизация с БД.
* **HTTP Methods:** Настроен роут с методом **PATCH/PUT** для корректного обновления ресурса согласно стандартам REST.
* **User Experience:** Формы предзаполнены текущими данными категории для удобства пользователя.
* **Data Persistence:** Использование метода `update()` модели Eloquent для сохранения изменений.

### 🗑 Шаг 9: Удаление данных и Soft Deletes (CRUD - Delete)
* **Destruction Logic:** Реализован `DestroyController` (Single Action) для безопасного удаления категорий.
* **Database Evolution:** Создана и запущена миграция `add_column_soft_deletes_to_categories`, добавляющая поле `deleted_at`.
* **Data Safety:** Интегрирован трейт `SoftDeletes` в модель `Category`. Теперь при удалении запись не исчезает физически, а помечается как удаленная, что позволяет восстановить данные при необходимости.
* **UI Action:** В список категорий добавлена функциональность удаления через скрытую форму с использованием метода **DELETE**.

### 🏷 Шаг 10: Масштабирование — Полный CRUD для Тегов
* **Consistency:** Реализован аналогичный архитектурный паттерн (Single Action Controllers) для управления тегами.
* **Feature Set:** Полный цикл обработки данных:
    * Просмотр списка и детальной информации.
    * Создание и редактирование через формы.
    * Безопасное удаление с поддержкой **Soft Deletes**.
* **Code Reusability:** Эффективное использование мастер-шаблонов и компонентов AdminLTE для унификации интерфейса админ-панели.

### 📝 Шаг 11: Полнофункциональный CRUD Постов и работа с Media
* **Advanced CRUD:** Реализована сложная логика управления постами, включая привязку категорий и тегов.
* **File Management:** Настроена загрузка изображений (превью и контент) с использованием фасада `Storage`. Реализовано хранение в `public` диске и генерация ссылок.
* **Complex Relationships:** Внедрена синхронизация тегов (Many-to-Many) через метод `attach()` / `sync()` в контроллере.
* **UI/UX Enhancement:** * Интегрирован **Summernote** (WYSIWYG-редактор) для удобного форматирования текста постов.
    * Подключен **Select2** для удобного мульти-выбора тегов с поиском.
* **Data Integrity:** Добавлена поддержка **Soft Deletes** для таблицы постов.

### 🏗 Шаг 12: Рефакторинг и внедрение Service Layer
* **Architectural Pattern:** Внедрен слой сервисов (**Service Layer**) для декомпозиции бизнес-логики.
* **Refactoring:** Логика создания и обновления постов перенесена из контроллеров в специализированный класс `PostService`.
* **Thin Controllers:** Контроллеры `StoreController` и `UpdateController` теперь отвечают только за прием данных и возврат ответа, делегируя всю работу сервису.
* **Maintainability:** Данный подход позволил избежать дублирования кода при обработке изображений и синхронизации тегов, сделав код более чистым (DRY - Don't Repeat Yourself).

### 🛠 Шаг 13: Локализация и кастомизация валидации (UX)
* **Validation Customization:** В классах `FormRequest` переопределен метод `messages()` для вывода понятных пользователю ошибок на русском языке.
* **Consistency:** Кастомные сообщения внедрены для всех модулей: Посты, Категории и Теги.
* **User Experience:** Теперь при некорректном заполнении форм (отсутствие заголовка, неверный формат изображения и т.д.) пользователь получает четкие и локализованные инструкции вместо стандартных системных сообщений.

### 👥 Шаг 14: Управление пользователями и правами доступа
* **User CRUD:** Реализован полный цикл управления пользователями (создание, просмотр, редактирование, удаление).
* **Security:** Внедрено хэширование паролей с использованием фасада `Hash::make` (алгоритм bcrypt) для обеспечения безопасности хранения учетных данных.
* **RBAC (Role-Based Access Control):** * Создана миграция для добавления колонки `role` в таблицу `users`.
    * Определены константы ролей (например, Admin, Reader) для разграничения прав доступа.
* **Form Logic:** Настроены специализированные запросы для валидации данных пользователя, включая проверку уникальности Email.

### 🛡 Шаг 15: Безопасность и Middleware (Access Control)
* **Authentication:** Подключен стандартный механизм авторизации и регистрации пользователей. Все административные маршруты защищены посредником `auth`.
* **Custom Middleware:** Разработан собственный компонент `AdminMiddleware` для проверки роли пользователя.
* **Route Protection:** Реализована многоуровневая защита:
    1. Проверка авторизации (`auth`).
    2. Проверка прав администратора (`admin`).
* **Kernel Configuration:** Новый посредник успешно зарегистрирован в `Http/Kernel.php`, что позволяет использовать его как в группах маршрутов, так и индивидуально.

### 📧 Шаг 16: Автоматизация и Email-уведомления
* **Password Generation:** Внедрена автогенерация безопасных паролей с использованием хелпера `Str::random(10)`.
  * **Mailables:** Создан класс `PasswordMail` для обработки почтовых отправлений `php artisan make:mail User/PasswordMail -m mail.user.password`.
* **Markdown Templates:** Разработан стильный шаблон письма с использованием Blade-компонентов `<x-mail::message>`.
* **Data Transfer:** Реализована передача динамических данных (сгенерированного пароля) в почтовый шаблон.

### 🛡 Шаг 17: Верификация Email (Email Verification)
* **Identity Assurance:** Внедрен механизм подтверждения почты для новых пользователей.
* **Model Configuration:** Модель `User` расширена интерфейсом `MustVerifyEmail`.
* **Routing:** В `Auth::routes()` активирован параметр `verify => true`, что автоматически включает необходимые маршруты для подтверждения.
* **Event Dispatching:** В контроллер создания пользователя в админке добавлена генерация события `Registered($user)`, что инициирует автоматическую отправку письма с подтверждением.
* **Development Environment:** Настроена интеграция с сервисом **Mailtrap** для безопасного перехвата и тестирования исходящих писем в среде разработки.
